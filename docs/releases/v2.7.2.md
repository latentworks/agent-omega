# v2.7.2 — deterministic terminal task-quality candidate

## What changed

- A new engine terminal hook receives a stopped assistant response and its
  exact parent identity only in the loader-attested task-quality slot.
  Approval-bound final candidates stay private until that hook selects safe
  visible text; ordinary prose plans remain durable before plan review.
- A qualifying prose-only plan is automatically sent through the existing HSS
  or CRAP review route. It remains unapproved until the durable lifecycle says
  otherwise.
- A receipt-backed final artifact from the exact approval or artifact-review
  follow-up is automatically reviewed and can only close at
  `artifact-reviewed`.
- The Windows release bundle carries the managed task-quality plugin and
  safely refreshes it for recognized existing Agent Omega configurations.
  Personal providers, model choices, council data, memory, and user plugins
  are preserved; foreign OpenCode configurations are not changed.
- The Windows package includes exactly `engine/opencode.exe`; ignored local
  backup executables cannot enter a release bundle.

## Candidate verification

- App logic suite: 179/179 passing, including managed-plugin refresh,
  foreign-config refusal even when a foreign config has its own
  `task-quality` folder, terminal plan parent identity, and CRAP terminal
  artifact repair after a new settled receipt.
- Engine: full typecheck passed; the focused processor, attestation, and
  direct-dispatch suites passed 22/22. They prove that an approval-bound
  response never reaches the transcript before task-quality explicitly
  releases reviewed text, a missing release becomes fixed lifecycle feedback,
  and ordinary plugins cannot receive terminal evidence.

This record intentionally does not claim live-model or desktop-packaged proof
until the 2.7.2 engine asset and Windows candidate have completed the retained
Qwen3.6-35B campaign.
